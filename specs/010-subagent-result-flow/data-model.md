# Data Model: Subagent Result Flow

## Overview

The core data structures involved in routing the subagent's execution results back to the parent agent. Since this is an architectural decoupling, we are relying on existing models (`SessionResult`, `LoopEvent`) but standardizing their usage for inter-loop communication.

## Entities

### `SessionResult`

The return type of `runSession()`. This encapsulates the state of the session after loop execution completes.

```typescript
export type SessionResult = 
  | { status: "ok"; message: Message.WithParts }
  | { status: "error"; error: unknown; message?: Message.WithParts }
  | { status: "aborted" }
```

- **status**: Indicates the final state of the subagent loop.
- **message**: The final assistant message generated by the loop.
- **error**: The error encountered during loop execution (if any).

### `msgsBuffer` (Parent Agent)

The in-memory array of `Message.WithParts` that represents the ongoing conversation of the parent agent. When a subagent completes successfully, its final message from the `SessionResult` is appended directly to this buffer, rather than being read from the database.

## State Transitions

1. **Parent Loop -> Subagent Call**: The `streaming-tool-executor` or `queryLoop` initiates a subagent task by calling `SubagentRunner.run()` (which wraps `runSession`).
2. **Subagent Execution**: The child loop executes normally, emitting SSE events and persisting to its checkpointer.
3. **Subagent Completion**: The child loop terminates and returns a `SessionResult`.
4. **Result Processing**: 
    - If `ok`: The parent extracts `childResult.message` and handles it.
    - If `error` / `aborted`: The parent handles the failure state locally.
