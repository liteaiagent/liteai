# Implementation Plan: Plan Mode Lifecycle

**Branch**: `013-plan-mode-lifecycle` | **Date**: 2026-05-19 | **Spec**: [spec.md](./spec.md)

**Input**: Feature specification from `specs/013-plan-mode-lifecycle/spec.md`

## Summary

Rewrite the plan mode lifecycle from a state-machine toggle to a blocking subagent-driven architecture. `plan_enter` spawns a dedicated plan subagent, gates the root session to read-only, and returns the plan artifact. `plan_exit` provides a single approval gate, restores write permissions, and stores the plan. The `PlanModeState` interface drops deprecated fields (`active`, `workflowType`) in favor of `planSessionID`.

## Technical Context

**Language/Version**: TypeScript (Bun runtime)

**Primary Dependencies**: `@opentelemetry/api` (tracing), `zod` (validation), `effect` (permission layer), `ai` (Vercel AI SDK)

**Storage**: SQLite (drizzle-orm) + in-memory registry (`PlanModeStateRef`)

**Testing**: `bun test` (Bun native test runner, Jest-compatible API)

**Target Platform**: Server-side (multi-tenant HTTP/SSE backend)

**Project Type**: Monorepo (`packages/core`, `packages/cli`, `packages/web`, `packages/sdk`)

**Performance Goals**: Non-blocking event loop, minimal memory per session, KV cache reuse for subagent

**Constraints**: Breaking interface changes with event-level backward compatibility (PlanModeState drops `active`/`workflowType`; `PlanStateChanged` event retains a derived `active` field for TUI consumers)

**Scale/Scope**: ~17 files touched across `core` + `cli` packages

## Implementation Phases

### Phase A: Data Model & State (Foundation)

**Files**: `plan-mode-state.ts`, `session/index.ts`

1. Update `PlanModeState` interface:
   - Remove `active: boolean`
   - Remove `workflowType: "interview" | "5phase" | undefined`
   - Add `planSessionID: SessionID | undefined`

2. Update `createDefaultPlanModeState()` to match new interface

3. Update `PlanModeStateRef.update()`:
   - Change transition detection from `prev.active !== this._state.active` to `prev.planSessionID !== this._state.planSessionID`
   - Update `PlanStateChanged` event payload to include `planSessionID` and derive `active` for backward compat

4. Update `Session.Event.PlanStateChanged` payload schema:
   - Add `planSessionID: SessionID.zod.optional()`
   - Keep `active` as derived field for TUI compat

### Phase B: Permission Hardening (Pre-requisite)

**Files**: `server/routes/session.ts`

1. Update `permission-mode` route handler (L550-562):
   - Replace `ref.get().active` check with `ref.get().planSessionID` check
   - Replace `active: true/false` mutations with `planSessionID` mutations

2. `permission/service.ts` — **NO CHANGES NEEDED**. The `plan` mode hard-deny at L191-195 already blocks all write operations. The plan subagent runs in its own session with independent permissions.

### Phase C: `plan_enter` Rewrite (Core Change)

**Files**: `tool/plan.ts`, `bundled/prompts/tools/plan-enter.txt`

1. Remove `interviewMode` parameter (FR-015)
2. Remove `Question.ask` approval gate (FR-014)
3. Remove `PlanApprovalRequested` event from `plan_enter` (FR-014)
4. Remove workflow loading (`Bundled.miscPrompt("plan-workflow" | "plan-interview")`)
5. Replace `state.active` guard with `state.planSessionID` guard
6. Implement new blocking subagent flow:
   ```
   a. setPermissionMode(ctx.sessionID, "plan")
   b. Session.create({ parentID: ctx.sessionID })
   c. SessionPrompt.runSubagent({ agent: "plan", ... })
   d. Parse result → extract plan text from subagent output
   e. Update PlanModeStateRef: set planSessionID + planText
   f. Return { planFilePath, planText } to root agent
   ```
7. Error recovery: try/catch around runSubagent, restore permission mode on failure
8. **Note**: `runSubagent()` does not expose a timeout parameter. Timeout protection is delegated to the underlying LLM provider's request-level timeouts and the session's abort signal. See the "Subagent spawn timeout" entry in the risk register for current mitigations and planned enhancements.

9. Update `plan-enter.txt` tool description to reflect the new behavior (no approval gate, spawns subagent)

### Phase D: `plan_exit` Update (Approval Gate)

**Files**: `tool/plan.ts`, `bundled/prompts/tools/plan-exit.txt`

1. Replace `state.active` guard with `state.planSessionID` or `planText` guard
2. Keep plan write-to-disk, `PlanApprovalRequested` event, and `Question.ask` approval dialog
3. On approval:
   - Call `SessionPrompt.setPermissionMode(ctx.sessionID, "default")`
   - Update state: clear `planSessionID`, set `planText`, reset `turnsSincePlanReminder`
   - Remove `workflowType: undefined` (field deleted)
4. On rejection: no state mutation (preserves plan mode — unchanged behavior)

### Phase E: Downstream State Consumers (Ripple Effects)

**Files**: `plan-reminder.ts`, `stop-drift.ts`, `query.ts`, `loop.ts`, `filter.ts`

1. **plan-reminder.ts**: Replace `planModeState.active` → `planModeState.planSessionID !== undefined`
2. **stop-drift.ts**: Replace `planState.active` → `planState.planSessionID !== undefined`
3. **query.ts**: Replace `planModeState.active` → `planModeState.planSessionID !== undefined` at L263 and L563
4. **loop.ts**: No changes needed — `createDefaultPlanModeState` and `PlanModeStateRef` lifecycle calls don't reference `active` directly
5. **filter.ts**: Replace `task` in `disallowedTools` with `agent` (already done in P1 — verify only)

### Phase F: Plan Agent Config Update

**Files**: `bundled/agents/plan.md`

1. Remove `write` from `disallowedTools` (plan agent needs to write plan file)
2. Replace `task` with `agent` in disallowedTools (if not already done in P1)
3. Remove `plan_exit` from disallowedTools (already filtered by `ALL_AGENT_DISALLOWED_TOOLS`)
4. Update prompt to instruct the subagent to:
   - Write the plan to disk using the `write` tool
   - Return the full plan text as its final response text
   - Include the plan file path in the response

> **Write Scoping Risk**: The `write` tool is unrestricted at the tool level. Scoping is enforced only via the agent prompt (instructs write only to `planFilePath`), which is **not a reliable security boundary** — a malicious or confused plan subagent could write to arbitrary paths outside `planFilePath`. The plan subagent runs in an isolated child session, which limits blast radius to its own session context, but does not prevent filesystem writes. **Mitigations** (deferred to a future phase): (1) runtime path whitelisting in the write tool's permission model to restrict writes to `planFilePath`, (2) enforce write-tool permission checks at the tool layer rather than relying on prompt instructions, (3) consider session-level filesystem sandboxing for subagents. See the "Write tool prompt-only scoping" entry in the risk register.

## Risk Register

| Risk | Impact | Mitigation |
|------|--------|------------|
| Subagent spawn timeout | Root agent stuck in read-only indefinitely | `plan_enter` error recovery restores default permission mode; LLM provider request timeouts apply; session abort signal propagates cancellation. **Future**: add `AbortSignal.timeout()` wrapper around `runSubagent()` for tool-level timeout enforcement |
| `PlanStateChanged` event payload change breaks TUI | TUI plan mode indicator stops working | Keep `active` as derived field in event payload for backward compat |
| Plan agent writes to wrong path | Plan file not found by `plan_exit` | Plan file path is deterministic (`Session.plan()`) — agent instructed to use it; scoping is prompt-level, not runtime-enforced |
| Write tool prompt-only scoping | A malicious or confused plan subagent could write to arbitrary filesystem paths outside `planFilePath`, since the `write` tool has no runtime path restriction — scoping relies entirely on the agent prompt | (1) Add runtime path whitelisting/allowlist to the write tool's permission checks, restricting plan subagent writes to `planFilePath` only; (2) enforce write-tool permission checks at the tool layer; (3) isolate subagent sessions with filesystem sandboxing; (4) scheduled for a future hardening phase |
| Concurrent `plan_enter` calls race | Duplicate plan sessions | `planSessionID` guard + root-agent-only check prevent this |
| CLI `prePlanPermissionMode` logic breaks | Permission mode not restored after plan mode | Permission restoration now handled by core (`setPermissionMode("default")` in `plan_exit`), CLI logic becomes redundant but harmless |
